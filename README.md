# auto-diet 웹사이트

인원, 예산, 운영 요일만 입력하면 식단을 자동 생성하는 급식소 관리 서비스

---

## 📋 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **개발 기간** | 2025.12 |
| **프로젝트 유형** | 개인 프로젝트 |
| **핵심 기술** | Next.js, TypeScript, Supabase, TanStack Query, Tailwind CSS |
| **배포 링크** | [바로가기](https://auto-diet.vercel.app/) |

## 🎯 프로젝트 목적

"반복되는 식단 고민, 시스템이 짠다."

영양사가 매번 식단을 짜고 영양·원가를 따로 계산하는 반복 업무를 없애기 위해 개발한 식단 자동화 서비스입니다.<br>
식약처 영양 DB와 농수산식품유통공사 시세 데이터를 연동해 식단 생성부터 영양·원가 확인까지 자동화합니다.

| 구분 | 내용 |
|------|------|
| **기존 방식** | 영양사가 직접 메뉴를 선정하고, 영양 정보와 식재료 원가를 별도로 조회·계산 |
| **본 서비스** | 인원·예산·운영 요일을 입력하면 식단을 자동 생성하고, 영양 정보와 실시간 원가를 자동 반영 |
| **효과** | 반복적인 식단 작성·원가 계산 업무를 줄이고, 예산 초과·메뉴 중복 없는 식단을 빠르게 확보 |

---

## 🖥️ AI 활용 개발 방식

이 프로젝트는 **바이브 코딩(Vibe Coding)** 방식으로 개발했습니다.<br>
AI에게 원하는 결과를 설명하고 실행 결과를 보며 진행하는 방식입니다.

**사용 도구:** ChatGPT, Lovable, Cursor

**개발 흐름:**

1. ChatGPT와 기능 요구사항 및 기술 방향 논의·구체화
2. ChatGPT로 최종 프롬프트 작성
3. Lovable에서 초기 코드 생성
4. Lovable에서 동작 확인 및 수정 후 GitHub push
5. 로컬 clone 후 Cursor에서 세부 기능 구현 및 코드 수정
6. 실행하여 기능 동작 여부 확인
7. 오류 발생 시 문제 상황을 설명하고 수정 요청
8. 정상 작동할 때까지 반복 수정
9. 정상 동작 확인 후 성능 최적화 요청
10. 최종 동작 확인 후 배포

**개발자 역할**: 기능·기술·성능 요구사항 결정, 비즈니스 로직 설계, AI 결과물 검증 및 동작 판단<br>
**AI 역할**: 코드 작성, 오류 수정, 구조 설계 제안

## 🔑 핵심 구현 내용

### 1. 하이브리드 식단 생성

식단 생성을 즉시 수준으로 단축하기 위해, 식단 구성과 영양 보강을 분리하는 하이브리드 방식을 설계했습니다.

- 검증된 하드코딩 데이터로 식단 구성 후 즉시 화면 표시
- 식약처 API → 영양 정보, 한국농수산식품유통공사 API → 원가 정보 순차 보강
- 메뉴 데이터에 `weight` 값으로 선택 확률 조정 (인기 반찬 weight: 30, 드문 반찬 weight: 4)
- 특별식(일식·중식·양식) 설정 시 해당 카테고리 전용 반찬 풀로 전환, 부족하면 공용 풀로 폴백
- 끼니 생성마다 예산에 가장 근접한 조합을 최대 18~120회 시도하고, 허용 오차(예산의 3%) 달성 시 조기 종료

### 2. 영양·원가 데이터 보강

식약처 API 호출량 제한(1,000건/일)에 대응하는 다단계 캐싱과 급식소 기준 데이터 변환을 구현했습니다.

- API 호출 우선순위: LocalStorage 캐시(24시간) → Supabase DB → 식약처 API → 하드코딩 fallback
- 협력적 캐싱: 첫 호출자가 DB에 저장하면 이후 사용자는 API 없이 즉시 반환
- 영양 보강은 주간→일→끼니 3단계 `Promise.all`로 병렬 처리
- 식약처 API의 100g 기준 영양 정보를 카테고리별 급식소 1인분(밥 210g, 국 250ml, 육류 80g 등)으로 환산
- 탄단지 비율(탄수화물 45-65%, 단백질 10-35%, 지방 20-35%)이 권장 범위에 드는지 별도 검증

### 3. 메뉴 중복 방지 & 식재료 충돌 차단

같은 날부터 월간 범위까지 5단계 중복 방지를 적용하고, 식재료 수준의 충돌도 별도로 차단합니다.

- 같은 날 → 인접 일자 → 같은 주 → 인접 주차 → 월간(4주) 순으로 `isMenuAllowed()` 순차 검사
- 밥·국·김치류는 매일 등장이 자연스러워 중복 체크 제외
- 다중 주차 생성 시 `prevWeekMenus·nextWeekMenus·monthMenus`를 컨텍스트로 전달해 주차 경계를 넘는 중복도 차단
- 김치찌개 생성 시 배추·파·열무김치 자동 제외, 두부·순두부 등 같은 주재료 그룹을 한 끼에 중복 배치 차단

### 4. AI 영양 균형 분석

주간 식단 전체에 대해 AI API를 호출해 영양 균형을 분석하고 개선 제안을 제공합니다.

- Supabase Edge Functions를 통해 서버리스로 OpenAI API 호출, 클라이언트에 API 키 노출 없이 처리
- 칼로리, 단백질, 나트륨, 다양성 지표별 상태를 개별 분석하여 `overallScore(0~100)` 산출
- 우선순위(high / medium / low)가 부여된 개선 제안 목록과 잘된 점 목록을 함께 반환
- 영양 균형, 메뉴 다양성, 원가 효율성, 알레르기 안전성 4개 관점에서 종합 평가

### 5. AI 대체 메뉴 추천

알레르기 포함 메뉴가 감지되면 AI가 조건에 맞는 대체 메뉴 3가지를 추천합니다.

- 알레르기 설정 항목이 포함된 메뉴 자동 감지 → "대체 추천" 버튼 노출
- OpenAI API에 알레르기 회피, 유사 영양가, 예산 조건을 프롬프트에 명시해 호출
- 카테고리(밥 / 국 / 반찬)별 맥락을 반영한 3가지 대안 반환, 각 항목에 추천 이유 포함
- 사용자가 선택한 대안으로 원래 메뉴 즉시 교체

### 6. Google Analytics

서비스 이용 패턴과 사용자 행동 데이터를 수집해 기능 개선 의사결정에 활용합니다.

- 페이지별 체류 시간, 이탈률로 식단 생성 → 영양 분석 → 메뉴 변경 사용 흐름 파악

- 기기, 브라우저 분포 확인 → 모바일/데스크탑 최적화 우선순위 결정

## 🔧 트러블슈팅 / 개선 경험

### 기술 선택

- **AI 식단 생성 비용 문제** : 초기에 OpenAI API로 메뉴 조합을 추천받는 방식을 검토했으나, 1인 1회 사용은 부담이 크지 않지만 사용자가 늘어날수록 API 호출량이 증가하고 재생성·부분 교체까지 고려하면 실서비스 운영 비용이 엄청 커질 수 있는 구조였습니다. 하드코딩 메뉴 데이터를 기반으로 중복 방지·예산 근접·weight 확률 규칙으로 조합하는 방식으로 전환하고, AI는 영양 균형 분석과 알레르기 대체 추천에만 활용하도록 역할을 분리했습니다. AI가 좋다고 무조건 넣는 것이 아니라, 비용과 효과를 따져 AI가 실질적인 가치를 더하는 곳에만 선택적으로 활용해야 한다는 것을 배웠습니다.

### API

- **식약처 API 호출량 초과 위험** : 식약처 API 무료 호출량이 1,000건/일로 제한되어 있어 매 식단 생성마다 호출하면 한도 초과 위험이 있었습니다. LocalStorage에 24시간 캐시를 구성하고, 로그인 시 Supabase DB로 마이그레이션해 첫 호출자가 저장하면 이후 사용자는 API 없이 즉시 반환되는 협력적 캐싱 구조를 설계했습니다. 실패 시 하드코딩 기본값으로 fallback해 서비스가 중단되지 않도록 처리했고, 외부 API 의존도를 낮출수록 서비스 안정성이 높아진다는 것을 확인했습니다.

- **공공 API 간헐적 실패** : 공공 API 호출 시 간헐적 실패와 429 Rate Limit 오류가 발생해 식단 보강 작업이 중단되는 경우가 있었습니다. 지수 백오프(`initialDelay × 2^attempt ± 20% 지터`)와 토큰 버킷 Rate Limiter를 구현하고, 429 응답 시 Retry-After 헤더를 우선 적용하도록 처리했습니다. 외부 API 연동에는 재시도 로직과 Rate Limiter를 기본으로 설계에 포함해야 안정성을 보장할 수 있다는 것을 배웠습니다.

### 디버깅

- **다중 주차 날짜 불일치** : 주차 추가·삭제·순서 변경·시작일 변경 시 날짜와 주차 번호가 맞지 않는 문제가 발생했습니다. `normalizeWeekPlans()`를 구현해 배열 순서 기준으로 날짜와 `weekNumber`를 일괄 재계산하고, 모든 상태 변경 이벤트에서 동일한 정규화 로직이 실행되도록 통일했습니다. 상태 변경 경로가 여러 개일 때는 각 경로마다 처리하는 것보다 단일 정규화 함수로 수렴시키는 구조가 일관성 유지에 훨씬 효과적이라는 것을 깨달았습니다.

- **백그라운드 작업 경쟁 조건** : 사용자가 새 식단을 생성하거나 로그아웃하면 이전 백그라운드 보강 작업이 완료되면서 새 화면을 이전 식단으로 덮어쓰는 버그가 발생했습니다. `enrichJobIdRef`를 이용한 취소 토큰 패턴을 도입해 새 작업 시작마다 ID를 증가시키고, 완료 시점에 현재 ID와 비교해 결과 반영 여부를 결정하도록 처리했습니다. 비동기 작업이 경쟁하는 환경에서는 완료 시점에 결과를 반영해도 되는지 검증하는 취소 토큰 패턴이 필수라는 것을 확인했습니다.

- **모바일 인앱 브라우저 로그아웃 오작동** : 카카오톡 인앱 브라우저에서 `signOut()` 이후 상태 반영이 늦어, 로그아웃 후에도 이전 사용자 데이터가 잠시 남아있는 현상이 있었습니다. `router.replace()` + `router.refresh()`를 조합해 세션 상태를 강제 동기화하고, 로그아웃 직후 클라이언트 상태를 즉시 초기화하도록 처리했습니다. 인앱 브라우저 환경은 일반 브라우저와 세션 처리 타이밍이 다를 수 있어 상태 초기화를 명시적으로 보장해야 한다는 것을 깨달았습니다.

### 수익화

- **AdSense 승인 반려 반복** : 서비스에 광고를 붙이기 위해 AdSense를 신청했으나, 심사 신청 시 서비스 메인 도메인이 아닌 블로그 URL로만 신청 가능한 구조였고 블로그로 승인받더라도 메인 서비스 페이지에 광고가 노출되지 않을 가능성이 있었습니다. 도메인을 직접 구매해 티스토리와 연결한 뒤 정보성 글을 꾸준히 작성하고, 승인 후 도메인을 서비스에 재연결하는 방법으로 현재 약 3개월째 시도 중입니다. AdSense 승인 구조상 서비스 도메인 자체로 신청하기 어렵고 우회 경로가 필요하다는 점, 승인 기준이 명확하지 않아 장기 시도가 필요하다는 것을 체감하고 있습니다.

- **단계적 수익화 설계** : 토스페이먼츠 연동 코드를 미리 구현해두었으나 초기부터 유료 결제를 요구하면 신규 유입이 막힐 것으로 판단했습니다. 서비스는 완전 무료로 오픈하고, AI API 비용은 AdSense 광고 수익으로 충당한 뒤 사용자가 충분히 확보된 이후에 구독형 프리미엄 플랜을 단계적으로 출시하는 구조로 설계했습니다. 수익화는 진입 장벽을 최소화한 상태에서 단계적으로 전환해야 초기 사용자 확보와 지속 가능성을 함께 잡을 수 있다는 것을 확인했습니다.
